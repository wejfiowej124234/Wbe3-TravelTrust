**Gate 通过后的持续合规机制**（非一次性流程，否则门禁失效）：
- **参数版本锁定**：过门时锁定的 08-3 版本号与该 Gate 的 evidence bundle 一一对应；参数变更须走变更记录，若触及该 Gate 相关 key 须评估是否重新过该 Gate（见 W-PDP-SSOT-CONSISTENCY）。
- **Gate 证据不可撤销**：evidence bundle 不得删除或改写，仅可追加「作废说明」并留痕（如法律结论变更后旧结论作废）；撤销须留 workitem_id + 批准人 + 原因。
- **Gate 有效期与复检**：Gate 结论有效直至下列任一项触发：① **法律/监管环境重大变化**（首发司法域或关键业务地法规变更）② **08-3 与该 Gate 相关的 key 变更**（须同步 08-4 并评估复过门）③ **定期复检周期**（**强制 12 个月**；具体日期写 Runbook §6）。触发后须重新评估是否重新过该 Gate；**复检时必须产出新 evidence bundle 或留痕结论**（不得仅「建议」），否则视为门禁失效。**法律环境变化后 3 个月内须完成复评并留痕。**

## 表头定义

| 列 | 说明 |
|----|------|
| **workitem_id** | 唯一工单号（W-xxxx / W-A1-STRUCT / W-BOMB-nn） |
| **Gate** | Gate-1 合规 / Gate-2 密钥与资损 / Gate-3 仲裁与声誉 / Gate-4 黑天鹅 / Gate-5 可替代性 |
| **08节** | 对应 08 文档章节（A1、B1、D2、K、Q6 等） |
| **状态** | ⬜ 待定稿 / 🟡 部分 / ✅ 已闭合 |
| **Decision（写死结论）** | 二选一/三选一写死句，不可歧义 |
| **SSOT 落点** | 参数表(SSOT) / 对外口径包(PDP) / Runbook，至少一个 |
| **Owner（唯一）** | **须为唯一主责角色/责任人**（可附 co-owner）；不得仅写部门名（如「法务+运营」），否则延期无法追责、验收无法签字。填写示例：法务（张三）、运维（李四配合）。**主责离职时须有 backup owner（替代人）**，否则制度依赖单人；定稿时在工单或 Runbook 中指定 backup 或替代机制。**Owner 失联时**：须有**自动升级到 backup** 的触发条件与流程（如 T+72h 无响应则 backup 接替并留痕），写死于 Runbook 或工单说明，制度不能依赖单人可及性。**制度与执行一致**：Owner 与 backup 须**真实签署或邮件确认**；建议有**轮值或交接制度**；**至少一次 Owner 失联演练**（backup 接替流程），否则制度≠执行。 |
| **Evidence（证据产物）** | 须尽量填**可验证指针**（如 evidence/GO_YYYYMMDD/ 路径或 manifest.sha256），便于复核人一键验真；描述型证据（报告/模板/截图）须附路径或 hash。定稿版本号、签字、演练编号等可并列。 |
| **due_date** | 计划闭合日期（可选） |
| **depends_on** | 依赖的 workitem_id，谁卡谁一眼看清（可选） |

**定稿时 Owner 落人**：下表 Owner 列当前为**角色名**（如法务、运维）；**过门签字前须逐条替换为「角色（姓名或代号）」**，如 法务（张三）、运维（李四），否则该工单不得视为闭合。可先填「法务（定稿时落责任人）」等占位，签字前替换为真实责任人。

**CI 规则**：若 `git diff` 触及 08-3 中「关键 key 与 08-4 章节映射」表中的 param_key，且 08-4 文末**文档版本（CI 校验用）**未在本 PR 中变更 → **阻断合并**。部署脚本：合约/升级代理部署前校验 08-3 参数表版本或关键 key，未通过则阻断部署。

**CI 豁免机制**（豁免也必须留痕，否则不得放行）：① **允许豁免的条件**：仅当 08-2 工单明确标注「可豁免 CI 同步」或法务/架构书面豁免（如合规例外、紧急热修）；② **留痕格式**：PR 中必须填写 **08-2 workitem_id** + **豁免原因**（一句话）+ **批准人**（姓名或代号）；③ **证据**：豁免记录须入 evidence 或 08-3 变更记录，否则 CI 不得放行；④ 乱填或绕过 CI 视为门禁失效，发版前须补正。

**防参数外改（CI 防文档、还须防运行时）**：① 运维直接改 Backend config 或合约未走部署脚本时，**须有链上/Backend 参数读取校验**（部署脚本或定期 job 比对 08-3 与运行时参数，不一致则告警或阻断）；② **运行时参数校验日志**须留存（谁、何时、改了什么），便于审计「参数外改」；③ **SSOT 运行时校验**（企业级）：Backend **启动时**应校验或打印当前 SSOT 版本/hash（与 08-3 或约定版本一致），链上读取参数与 08-3 关键 key 比对；否则存在「文档正确、运行错误」的穿透风险；④ 未通过 SSOT 的变更视为违规，见 W-DRIFT-CI。
