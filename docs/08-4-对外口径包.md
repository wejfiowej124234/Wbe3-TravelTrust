## 2. 资金与托管边界

- 平台**不托管**用户资金；资金在链上 Escrow，用户自签交易。
- **多签成员行为准则**：（写死）多签不得单独裁量、所有动作必须引用 SSOT 参数；多签是否被认定为 fiduciary 以法务监管穿透分析为准，见 Runbook §10、§7。
- **多签与「共同控制人/事实托管」边界**（写死）：① **收益隔离**：多签成员不因执行 SSOT 动作而获得订单/用户资金收益分配；仅可收取明确约定的运维/治理报酬（若有），且不与单笔订单或用户资金挂钩。② **免责边界**：按 SSOT 与 Runbook 执行者，免责条款见 ToS；超出 SSOT 的裁量由执行方自行承担。③ **退出机制**：多签轮换与失联恢复见 Runbook §10；退出后密钥作废、新多签接替；无单点「终极钥匙」。④ **实质控制披露**（监管穿透）：多签成员**须来自不同组织**（或披露关联关系）；**不得由同一法人/同一控制方单独控制**；**若多签 3/5 或 2/3 全部为同一公司成员，则为单点控制、门限无意义**。存在内部关联时须在 Runbook §10 或治理文档中披露；企业级尽调会问股权穿透、实际控制人。**多签成员须提供「实质控制声明」**（股权/法人/控制关系无共同实际控制人），并**定期更新**（频率见 Runbook §10）；未声明或逾期未更新视为不合规。
- **事实控制边界说明**（写死）：平台**不托管**用户资金（资金在链上 Escrow、用户自签）；**多签具备 Pause/冻结能力**属于**运维与安全应急权限**，非对用户资金的日常控制；Pause 仅暂停链上转出、不转移资产所有权；恢复后资金仍按合约与裁决执行。监管问「是否具备事实控制能力」时：资金终态仅来自链事件与裁决执行，多签不能单方划转用户资金。**穿透监管问的是「是否具备改变资金流向的能力」**（非「是否实际控制」）；若多签可升级 Proxy 改资金流向、可改 disputeWindow 使资金更易流出、可改释放条件，则具备控制能力，属**监管定性问题**。**必须写死**：**Immutable Core 是否真正不可改**、**Proxy 是否可换 Implementation**、**是否存在超级管理员**—否则「Immutable」只是口号；落 01/02 与 Runbook §7。
- **多签权限矩阵**（写死，企业级必补）：须明确区分 **可改「运营参数」** 与 **不可改「资金终态逻辑」**。多签**可改**：Pause 白名单、allowlist、冷却期、证据保留期等**运营/安全参数**（见 08-3 与 Runbook §7）。多签**不可改**（或仅限 Immutable Core 外 Shell）：**资金释放终态逻辑**（如 Escrow 释放条件、裁决执行逻辑）、**单笔资金流向**；若 Proxy 可换 Implementation 或存在 admin/override，须写死**是否存在超级管理员、是否存在紧急 override**及边界。**多签权限矩阵**（一页表或 Runbook §7/§10）须写死，否则「Immutable Core」仅为概念、无权限约束图。见 Runbook §7、08-3 权限层级。
- **多签与公司级管理层**（写死）：多签是否承担**公司级 fiduciary 责任**、是否被视为「管理层」以**法务按首发司法域的监管穿透结论**为准；**谁是管理层**须在治理文档或 Runbook 中写死（公司结构问题，非仅门限）。多签执行 SSOT 动作不改变「技术协议 vs 金融服务商」的定性；定性由法务与监管沟通结论为准。
- **收入归集路径与收益控制**（写死，公司级）：**仲裁费、质押罚没、订单手续费、罚款、质押收益**的归集路径须写死：进入**运营账户 / 链上池 / DAO treasury** 三选一或组合，以及**是否可被分红**；未写死则监管会问「是否具备收益控制能力」并影响技术协议 vs 金融服务商定性。**企业级须补齐一页「收益流闭环图」（6️⃣）**：① **仲裁费→去哪？质押罚没→去哪？手续费→去哪？** 路径写死；② 是否进入公司账户、是否进 DAO、**是否可分红、是否可回购**—若可分红可能触发证券属性，须写死**无分红**或法务结论；③ **会计确认时间点**（仲裁费何时算收入、被罚质押何时算收益）；④ **资金托管证明**（不托管用户资金）。**企业级审计会要求一页收益流闭环图**；具体路径与会计归属见治理文档或 Runbook 定稿表；**不得**形成与「不托管」冲突的收益控制口径。**收益证券隔离证明（Securities Exposure）**（写死）：监管必问—① 仲裁费去哪 ② 质押罚没去哪 ③ 是否进公司账户 ④ 是否分红 ⑤ 是否可回购 ⑥ 是否与治理币价值绑定。**只要有收益共享结构即有 Howey Test 风险**。须形成**证券属性风险隔离声明**完整闭环：**无收益共享**（或法务结论的「不构成证券」）、**无分红、无与治理币价值绑定**；收益机制写死且与「不触发证券属性」声明一一对应，落治理文档或本节附录。否则收益机制完整但证券隔离未闭环。
- **会计确认点**（写死，企业级）：**仲裁费何时确认收入、被罚质押何时计入、退款是否冲减收入**等会计入账规则须在财务/治理文档中写死；**企业级须有会计确认 memo 或等效落点**（每类收入的确认时点、冲减规则）。08 系列定稿「机制」，会计规则由财务与审计定稿并可与 08-4 交叉引用。
- 可暂停原则：多签可触发 Pause，暂停期间用户资金不可转出；恢复条件见第 6 章。
- **协议层与运营层边界**：（写死）谁签字执行裁决、谁可 Pause、谁可轮换—见「协议层与运营层边界图」或一页说明；监管问「谁在执行」可一句话回答。
- **权限层级**：（写死）常规多签受 paramFreezeDays、paramChangeMaxPer30d 约束；**紧急多签**仅限 Runbook §7 列明场景（Pause、OFAC/监管执行、安全事件），可豁免冷却期/参数冻结期，**不可**改法务口径 key；无「超级多签」或「终极钥匙」。
- **Immutable Core 与终极控制路径**（写死，企业级必答）：若声称 **Immutable Core 不升级、Shell 可升级**，须写死：**Proxy 是否能换 Implementation**、**是否存在 admin key**、**Timelock 是否可被绕过**；**是否存在超级管理员、是否存在紧急 override**（若有则边界写死）。企业级审计必问「有无终极控制路径」—哪怕理论存在也须写死并落 Runbook 或 01/02 合约文档，否则「不可升级」仅为叙事。
- **紧急 override 与终极控制路径图（1️⃣）**（写死，企业级必答）：审计会问—**是否存在任何 emergency override？是否存在 timelock bypass？是否存在 admin slot 可替换？** 若存在任一，须写死：**谁触发**、**是否链上可见**、**是否需要二次确认**、**是否写入 08-4**；若不存在则写死**无**并落 01/02 与 Runbook §7。**企业级须能提供「终极控制路径图」**（一页图：从触发人→链上/链下动作→是否改资金终态）；**画不出来 = 风险**，须由架构/法务定稿并存 08-4 引用或 Runbook §7/evidence。
- **Escrow 资金路径数学封顶（链上层 1️⃣）**（写死，企业级必答）：Escrow 是否存在 **admin override、upgrade hook、emergency withdraw、delegatecall 可换逻辑**—若存在任一「理论上可改变资金流向」的代码路径，监管结论为 **governance-controlled contract，非 immutable settlement layer**。须写死：**哪些逻辑不可升级**、**哪些升级需要 timelock**、**哪些升级需要 N-of-M 多签 + timelock**、**是否允许改变历史订单逻辑**（若允许=可升级金融系统，非不可篡改协议）。落 01/02 合约文档、Runbook §7；由架构定稿并可与「终极控制路径图」合并。

## 3. 仲裁是什么

- **能裁什么**：（写死）订单履行争议、服务开始/未开始证明、证据充分性与权重适用。
- **不能裁什么**：（写死）线下人身伤害责任归属、刑事认定、非协议范围内的旅行体验质量。
- **服务开始时间的判定源**（写死，防链时间与现实/时区/篡改冲突）：以**证据时间戳 + 服务器时间**为准（与 08-3 serviceStartTimeSource 一致）；链上 **disputeWindow 以 block 为界**。用户设备时间不可单独采信。**时间双轨冲突时的优先级**（写死）：**裁决侧**（服务开始时间、证据时间）以 **backend_evidence + 可验证时间戳**为准；**争议窗口开闭**以**链时间（block）**为准；二者冲突时**窗口边界以链为准、裁决所采时间以 08-3 serviceStartTimeSource 为准**，写死于 Runbook §11，防时区/时间回滚攻击。**企业级须补齐**：① **时间优先级矩阵**（block vs 证据时间 vs 服务器时间—冲突时谁为准）写死于 Runbook §11；② **时间回滚检测机制**（检测到服务器时间回滚时的处置）；③ **服务器时间不可篡改证明方式**—**服务器时间是否可被人为改写**须写死（须不可人为改写）；**是否有 NTP 审计**、**是否有时间回滚检测**须写死；采用 NTP/时间戳公证/签名由运维/架构定稿并留 08-3 或 Runbook；④ **冲突决策流程图**可引用 Runbook §11 或独立一页，否则仲裁层存在时间攻击面。若服务开始时间来自 backend，**是否存在人为调整风险**须写死并缓解。
- **基本原则**：证据权重表、证据缺失默认裁决（退款/放款/按比例写死）；裁决签名可追溯、双人审批。
- **仲裁费用途与逆向激励防护**（写死）：仲裁费（arbFeeBase/arbFeeCap/arbFeeMultiplier）**不得进入平台营收或与争议数量挂钩**；用途须写死于治理文档：**仅用于运营与合规成本**（如证据存储、合规审计），或进入**公共池/治理金库**（不进入运营账户利润）；**不得**形成「鼓励争议」的隐性激励。定稿时在 08-3 或 Runbook 注明具体流向。
- **服务开始时间防篡改**（写死）：采用 **evidence 时间戳 + 服务器时间** 时，须**时间戳可验证**（如服务器时间签名、或 NTP 可信源/时间公证）；用户设备时间不可单独采信。具体实现见 Runbook §11 或 08-3 变更记录，避免仲裁时间可被攻击。
- **透明边界**：按月披露裁决统计（匿名化）；单案细节不公开。**裁决一致性指数**（写死）：是否对外披露**裁决偏差/一致性指数**须写死（如按仲裁员/地区 anonymized 偏差指标）；企业级要求透明度可引用本节或治理文档，未写死则透明度不足。**声誉洗白防护（12️⃣）**（写死）：若导游**接小单积累声誉再接大单违约**，须写死：**声誉权重是否随订单金额变化**（建议金额加权或大单违约权重更高）、**是否存在违约永久或长期标记**、**是否存在「声誉洗白」可操作路径**；由产品/风控定稿落 03 或 Runbook，否则存在声誉操纵风险。
- **证据删除与司法协助**（写死）：保留期（如 1095 天）届满后可删原文件；**hash + receipt 永久保留**，可向司法机构提供以证明「曾存在、何时、何人」；不保留可逆原文件。删除后司法协助策略见 Runbook §9。**证据链攻击（8️⃣）**（写死）：企业级会问—**证据是否可被批量删除？Hash 是否可被 GDPR 删除要求覆盖？司法命令是否 override 区块证据？存储是否集中在单一云？若 S3 被封，仲裁还能进行吗？** 须写死：证据批量删除策略（若有）、Hash 与 GDPR 冲突时的法律 override 流程、司法命令与链上证据优先级、**证据存储多地域/多云**（单云不可用时的 fallback）；否则数据层存在单点与合规攻击面。落 Runbook §11、隐私政策。**证据链司法级要求**（写死）：hash 须可**单向证明**存在与内容；**时间戳须防篡改**（非仅 server timestamp）；须有**独立时间锚**（如链上锚或可信时间公证）及**支持司法验证**的方式，否则跨境司法场景可能被质疑。见 Runbook §11。**GDPR 与链/hash 冲突**（写死）：**被遗忘权**与「原始可删、hash 永久保留」须协调；须明确 **hash 是否构成个人数据**（若构成则删除/匿名化策略须在隐私政策写死）；EU 用户场景下 hash 保留的法律依据与范围须由法务定稿。**若司法要求彻底删除（包括 hash）**：须写死 **hash 是否可脱敏**、**是否存在法律 override 机制**（如合法司法令要求删除时的处理流程）；否则合规与司法协助冲突。见隐私政策与 Runbook §9。
- **数据主权与跨境**（写死）：证据存储若**跨境传输**或涉及 EU 用户，须满足**首发司法域与业务地数据合规**（如 EU GDPR 传输限制、本地化要求）；是否 EU 数据合规、传输路径与法律依据须在隐私政策或 Runbook 中写死，否则不得在相关地区开展争议业务。**企业级须写死**：**数据主权分布结构**（数据落在哪些司法域）、**DPA（数据处理协议）**、**法律适用地**；**谁是 data controller**（数据存储在美国/欧洲/亚洲等跨境场景下的控制者）须在隐私政策或 Runbook 中写死，旅游为强线下监管行业，监管必问。
- **仲裁系统完全失效时的 fallback**（写死）：若仲裁系统**完全不可用达 30 天**（无裁决、资金卡住），按**默认裁决**执行：**退款优先**（与 08-3/03 缺失证据时默认一致）；具体触发条件与执行流程见 Runbook §11。无 fallback 不上线。

*工单：W-D2-EVID、W-BOMB-02*

## 4. 裁决统计口径

- **按月披露**：（写死）裁决类型占比、游客/导游支持比例、平均处理时长、按地区/金额区间脱敏后的件数；单类案件数 ≥ k 才披露（k 匿名），否则合并为「其他」。
- **不披露**：当事人身份、单笔金额、具体证据内容。
- **披露机制作为攻击面**（写死）：**攻击者可通过批量制造小额争议、或故意触发默认裁决**操控统计或拉低一致性指数；须有缓解措施：**异常争议率告警/自动冻结**（见 Runbook §11）、**最小样本量（k 匿名）**、披露口径与攻击面说明写死，否则披露机制本身可能被滥用。
- **最小可执行口径（定稿前必须写死）**：① **披露字段清单**（哪些指标对外、哪些仅内部）② **匿名化阈值**（如单类案件数≥k 才披露、否则合并为「其他」）③ **脱敏规则**（地区/金额区间/时间粒度）④ **最小样本量（k 匿名）**（避免披露过多触发隐私/合规风险）。未写死则上线后会卡在：披露太少被质疑黑箱；披露太多触发合规。

## 6. Pause：何时暂停、用户能做什么、何时恢复

- **何时暂停**：满足阈值表条件并由多签执行；每次 Pause 发布解释模板。
- **暂停时用户能做什么**：（写死）仅查看订单与余额、可提交申诉材料、不可创建新单或执行链上转出。
- **何时恢复**：健康检查 + 对账通过 + 多签批准，且受冷却期限制。
- **优先级矩阵（Pause×OFAC）**：（写死）监管（OFAC/政府请求）> 黑天鹅（Pause）> 仲裁/常规。**Pause 期间 OFAC 命中用户**：（写死）【允许退出】。
- **黑天鹅与冲突优先级总表**（写死）：① **OFAC 命中 + 黑天鹅冻结 + Pause 三者同时触发**：执行顺序**固定**为 监管（OFAC/政府请求）→ 稳定币冻结披露/处置 → Pause（若仍需）→ 仲裁排队或按 Runbook §6/§11 延后；冲突矩阵与具体步骤见 Runbook §11。② **OFAC + 黑天鹅 + 仲裁进行中**：先执行监管 → 再 Pause → 仲裁延后。③ **参数冻结期 + 紧急多签**：紧急多签可豁免 paramFreezeDays（仅 Runbook §7 场景），见 08-3 权限层级。④ **reorg > finalityN**：自动暂停结算；人工确认后恢复，见 Runbook §11。

*工单：W-BOMB-04、W-PAUSE-OFAC-PRIORITY*

## 7. 前端可替代路线

- **至少 2 个入口**：（写死）官方 Web 前端 + 可自部署只读/开源构建（或文档化可替代入口）。
- **可验证发布**：每次发布提供 manifest + hash + 可复现构建说明。
- **去中心化路线图时间线**：（写死）仲裁可替换、前端可替代、执行器可轮换—每项须有可执行时间线或阶段（含「暂无计划」），无时间线不对外承诺。**可替代性阶段完成标准**须写死：每阶段「完成」的验收条件（如用户是否可绕开平台运营层、仲裁层是否可被第三方替代）；企业级会问「用户能否绕开平台」「仲裁是否可替代」，仅有方向无阶段标准=叙事与事实不一致风险。
- **仲裁员最低容量**：（写死）仲裁员人数低于 **3** 人（与 08-3 minArbitratorCount 一致；Runbook §8）时，暂停新争议受理；存量争议按既有 SLA 或延长处理；降级策略见 Runbook §8。**单一地区仲裁员占比**超过 Runbook 规定上限（如 80%）时告警或暂停新争议（防地域单点），见 Runbook §8。
- **仲裁员实质控制风险防护**（写死）：仲裁员**须避免结构性控制**—不得全部来自同一经济利益群体（如全部来自导游侧或平台长期合作方）。须有：**仲裁员利益来源披露制度**（与争议方的利益关系）、**收入结构透明度**（报酬不与单案结果挂钩）、**外部/独立仲裁员引入比例**（由治理或运营定稿写死本节或 Runbook §8）。否则制度存在「实质控制」盲区。
- **仲裁员激励隔离**（写死）：平台收入与仲裁员报酬**不得**与「争议数量」或「裁决结果」挂钩形成鼓励争议的激励；**仲裁员不得按案件数量计酬**、**不得按裁决效率付费**、**须与争议金额脱钩**（否则=制造争议或偏裁动机）。须明确：**固定报酬/固定津贴**、**与裁决方向无关**、**与争议数量无关**；**是否存在固定报酬+绩效上限**须写死（若有绩效则上限与脱钩规则写死）；写死于治理文档或 Runbook §8。否则会被认定为**经济激励冲突/偏裁激励**，企业级必问。
- **协议可独立运行性须可测试**（写死）：**当前阶段**协议**不能**独立运行（混合态）；若对外声称「可独立运行」前，**须产出真实测试报告**（**关停官方前端测试**、**关停仲裁/运营测试**、**仅链运行测试**至少一项可复现）。无测试报告 = 仅理念，不得对外承诺可独立运行。
- **无运营团队时协议能否继续运行**（写死，极端哲学问题）：若**运营团队消失**（多签/人工仲裁/Runbook 执行者均不可用）：**dispute 如何结束、Escrow 是否卡死**须写死结论。当前混合态下：无运营则争议无法裁决、Escrow 可能卡死，故**当前 = 混合治理平台**。**协议是否具备「自然终局能力」**须写死：若**仲裁员全部消失、多签全部消失**，是否存在 **自动 fallback**（如 30 天仲裁失效后默认裁决）、**最终资金自动路径**、**时间触发终结逻辑**；若不存在则协议不具备自然终局能力，须在 Runbook §6/§11 写死「无运营时的终局路径」或明确「当前无自然终局、依赖人工/多签」。见 Runbook §6 协议终止与 §11 仲裁失效 fallback。

*工单：W-Q6-FE、W-BOMB-10、W-DECENT-TIMELINE*

## 8. 合规红线与政府请求

- **制裁/OFAC 命中后**：（写死）【只读冻结】；申诉时限 = **30 天**（与 08-3 ofacHitPolicy 一致）。
- **政府请求（数据/冻结/修改规则）**：（写死）合法且符合首发司法域要求的政府请求按 Runbook 执行并留痕；超出范围的请求拒绝并记录；必要时仅关停前端入口，链上合约与资金逻辑不受单点关停影响。

*工单：W-A2-OFAC、W-BOMB-11*

- **责任边界（现实安全非协议责任）**：协议只处理资金与托管，不承诺线下人身安全、旅行体验质量、不可抗力导致的人身或财产损失；人身伤害/失踪/抢劫等非协议责任，协议动作限于暂停账户、证据保全、配合执法；具体边界见 Runbook §11 与法务条款。*工单：W-EXPECT-BOUND*

- **不托管下的协助义务**（写死）：**不托管**用户资金；在人身伤害、导游失联、地震等不可抗力场景下，**平台协助义务**限于：① 提供**紧急联系人机制**（见 Runbook §11 或 ToS）；② 证据保全与配合执法；③ 不承诺资金外的赔偿或救援。协助范围写死于 ToS；投资人/监管问询时口径一致。

- **多签串谋发现与监督**：（写死）串谋发现方式（链上报警/对账异常/外部举报）、是否设外部监督人、链上或运营侧报警—至少一项写死并对外可引用。*工单：W-MULTISIG-COLLUSION*
- **仲裁员重大舞弊处理**（写死）：仲裁员舞弊（如收受贿赂、故意偏袒、泄露证据）一经确认：**永久退出**+ 统计披露；**重大舞弊须按首发司法域要求通报执法机构**（流程见 Runbook §8）；企业级须有「重大舞弊处理流程」写死并留痕。

- **协议可独立运行性（定位）**：（写死）**当前阶段 = 混合治理态**：链上资金 + 多签执行 + 人工仲裁 + Pause + Runbook 存在，**不是**纯无许可协议；多签/仲裁/前端/运营全撤后协议当前**不能**独立运行。**未来阶段**：见第 7 章去中心化路线图时间线（仲裁可替换、前端可替代、执行器可轮换）。**叙事一致（协议还是平台）**：若对外声称「去中心化协议」，监管会问—**为何可以 Pause？为何可以人工裁决？为何可以改参数？**叙事必须一致：当前如实表述**混合治理态**，**不得**在 marketing 写成「完全去中心化」直至路线图达成；可 Pause/人工裁决/改参数与「去中心化」的边界须在第 1 章与本章写死，否则叙事偏差将导致监管与信任风险。*工单：W-PROTOCOL-STANDALONE*
- **制度性悖论（1️⃣ 结构冲突）**（写死，定性问题）：**去中心化叙事 vs 强控制机制**存在结构冲突—现有多签、Pause、仲裁、OFAC、升级、参数门禁等**强控制**。须写死：**若控制权集中**=你是「平台」不是「协议」；**若控制权分散**=**谁承担监管责任**须由法务按首发司法域写死（如运营主体承担、协议不可被直接诉）。**这是定性问题，非仅技术**；当前如实写死=**混合治理态、非完全去中心化**，监管责任归属见 ToS 与法务条款。落 Runbook §10、治理文档。
- **旅游行业监管属性**（写死）：旅游属**线下服务、人身风险、消费者保护高敏感行业**；监管穿透程度高于纯 DeFi，首发司法域与业务地旅行/消费者保护法规须在 08-1/法务落点中写死；不得低估旅游监管属性。
- **政府请求 + OFAC + 用户诉讼时的法律责任**（写死）：**谁承担法律责任、协议是否被法院认定为可执行主体**属**公司法律结构问题**，须由法务按首发司法域写死（如运营主体承担、协议不可被直接诉等）；冲突时责任归属与可执行主体口径见 ToS 与法务条款，与 08-4 一致。
- **终极组合场景（企业级最后一问）**（写死）：若**稳定币冻结 + OFAC 命中 + 仲裁 backlog + 多签成员之一被诉 + 用户集体诉讼**同时发生，**是否可防火墙隔离（A）还是形成证据链指向公司（B）**—取决于：收益路径、法律实体结构、多签成员结构、实际控制披露。**制度写强不等于公司结构文件齐全**；须有**公司结构/法律实体文件**（收益路径与法律实体对应、多签成员与法人隔离、实际控制披露）写死或定稿落点，否则企业级审计会要求补齐。见 Runbook 定稿表与 08-1 战略层风险。
- **若被认定为金融服务商**（写死）：若未来被认定需**牌照、AML、资本充足率**等，**系统是否可以承载**须由法务/产品评估并写死。**法律环境突变 30 天应对**与**存量订单处理策略**已写死于 Runbook §6；另须写死**「被认定为金融服务商时的应对路径」**（如申请牌照、关停、迁移司法域、业务收缩等）由法务定稿并落 Runbook 或治理文档，否则企业级会问「能否承受被监管认定为金融服务商」。监管典型判定点（**是否控制资金流、是否可暂停、是否可升级、是否可拒绝交易**）— 已在本章与 Runbook §7 写死边界；若存在则须写死为「运维/安全权限」且不可改资金终态逻辑。**事实金融服务判定四问（5️⃣）**（写死）：监管不会只问「是否去中心化」，会问—**是否控制资金流向？是否有收益控制能力？是否决定争议结果？是否可修改资金释放逻辑？** 若答案为「是」可能被认定为金融服务。须写死：**若被认定为 MSB 怎么办**（申请牌照/关停/迁移等）、**若被要求强制冻结怎么办**、**若被要求 KYC 全覆盖怎么办**；由法务定稿落 Runbook 或治理文档，否则 08-4 仅为理想口径。

## 协议终极边界声明与终局设计（企业级必答）

以下三件事未写死则视为**缺最后一层**：**协议终极边界声明**、**不可逆终局路径**、**控制权绝对封顶证明**。

**1️⃣ 终极 override 路径**（写死）：企业级会问—**是否存在任一单点或组合路径可改变资金释放规则？** 例如：Proxy 可升级 Implementation + Timelock 可缩短 + 多签可替换 + 紧急多签可 override 参数，若**组合起来可改变资金终态逻辑**，则 Immutable 只是制度、非事实。**必须能回答**：**是否存在任何单点或组合路径可改变资金释放规则？** 答案须为**绝对 NO**，或写死**唯一例外**并**封顶**（如仅限某类紧急、不可改释放规则）；否则仍为「事实控制结构」。落 01/02、Runbook §7、多签权限矩阵。**监管穿透模拟（Regulatory Kill Test）**（写死）：监管会问—**能否阻止某用户提现？** 若能→具备事实控制能力；若不能→OFAC 怎么执行？核心=**是否具备「改变资金流向」的能力**。若 Proxy 可升级资金逻辑→具备；若多签可 override 资金终态→具备；若 Immutable Core 真不可改→不具备。须在 01/02 或 Runbook §7 提供**不可逆结构图证明**（一页图或等价文档）：证明**无路径可改变资金流向**，或唯一例外写死并封顶；与终极控制路径图、Escrow 数学封顶一致。**无此证明=审计会要求补齐**。

**2️⃣ 多签是否实质集中（6️⃣ 公司穿透）**（写死）：监管会问—公司是否具备**实际收益控制能力、实际仲裁控制能力、实际暂停控制能力、实际升级控制能力**；**若 3/5 多签均为公司的人，门限是形式、非实质去中心化**。须写死：**多签是否跨法人**、**是否存在控制集中**、**是否公开披露结构**；禁止或披露同一投资方/董事会/实控人控制多签多数。由法务定稿落 Runbook §10 或治理文档。

**3️⃣ 协议死亡场景**（写死）：若**公司破产、多签成员消失、团队集体退出、域名被封、GitHub 删除**，协议是否**还能自然运行**？用户是否**可完成存量订单**？是否存在**自然终局路径**？若无写死，则仍为**平台型系统**。须在 Runbook §6 写死「协议死亡场景」下的终局路径（如 30 天仲裁失效后默认裁决、存量订单按 fallback 执行、用户资金最终处理逻辑），或明确「当前无自然终局、依赖人工/多签」及应对。

**4️⃣ 收益与证券属性**（写死）：**是否分红、是否利润分配、是否治理币分红**须写死。若**收益分配与协议利润挂钩或治理币分红**，可能触发 Howey。须写死：**无分红、无利润分配**，或**仅为功能型 token**（由法务定稿）；若有治理币须有法务 Howey 结论并落治理文档。

**5️⃣ 叙事与现实**（写死）：若**仲裁人工、多签集中、参数由公司控制、协议不可自然运行**，则实质为**带链的中心化平台**。该问题须自己敢回答：当前如实写死**混合治理态**、**非完全去中心化**；见第 1 章与第 8 章。

**6️⃣ 协议强制关闭（监管最后一问）**（写死）：**在什么情况下协议会被强制关闭？** 须能：① **列出至少三种明确触发条件**（如监管勒令、多签/仲裁不可恢复、稳定币永久冻结等）② **写出关闭流程** ③ **写出存量订单终局路径** ④ **写出用户资金最终处理逻辑**。若无写死则无真正「终局设计」。见 Runbook §6「协议强制关闭」节。

**7️⃣ 经济激励自洽**（写死）：**是否存在某角色通过制造争议反而盈利？** 若存在套利空间，长期必被利用。须写死：仲裁费/罚没/手续费/质押收益**不形成**「制造争议可盈利」的激励；若有套利空间须写死缓解或上限。见 08-4 第 3 章、Runbook §8。

**8️⃣ 治理币与宪法不可投票推翻（5️⃣ 治理捕获）**（写死）：若未来引入治理币，须写死—**谁初始分配、公司是否持有多数、是否可改仲裁规则/资金终态？** 若 governance 可改资金终态=证券+可控制金融产品。须写死：**治理可改什么、永远不可改什么**、**是否存在 constitutional guard**（宪法级条款不可投票修改）；**哪些条款须 supermajority、哪些永远不可升级**；由法务/治理定稿落治理文档或 08-4。**治理不可改资金终态（写死）**：**治理不可改资金终态**；若能改=协议不是不可篡改；若不能改=治理仅为「表决建议」。本条须在治理文档与 08-4 写死，不得留口子。

**9️⃣ 治理币设计边界与协议定位（抛盘/证券/不发币优先）**（写死）：**三种定位须选一写死**：**A. 纯业务协议（不发币）** — 无治理币抛盘问题，仅有信任与业务风险；**B. 业务 + 治理币** — 必然存在抛压（解锁/情绪），须结构性缓解；**C. 金融化协议** — 监管与证券风险最高。**当前推荐定位（写死）**：以现有架构（强调合规、SSOT、治理结构、风险封顶），**更适合不发币、走协议服务 + 合规扩张路线**；实质更接近 **Web3 结构化 SaaS**，非 DeFi Token 项目。**若未来发治理币**：须**纯治理、无收益权**— 无分红、无 treasury 分配、无 buyback、不与协议收益绑定；否则=证券风险极高。**结构性缓解（若 B）**：① **不绑定分红**（避免证券 + 收益驱动抛盘）；② **unstake 冷却期**（如 7～14 天）避免瞬间挤兑；③ **质押覆盖率上限**（total_exposure ≤ total_stake × safety_ratio）避免单波争议击穿；④ **参数变更冷冻期**（paramChangeMaxPer30d 已有）— 可加强为「参数修改触发冻结期、连续修改触发暂停治理权」。**两类非币「抛盘」须写死**：① **导游质押挤兑**：连续裁决偏向游客→导游亏损、质押被扣→导游集体退出并 unstake→订单骤降、质押池缩水、信誉系统崩塌（协议信任抛盘，比币跌更危险）；缓解=默认裁决平衡、连续偏向阈值、质押覆盖率与 worst-case 模型，落 08-3、Runbook §11。② **叙事/信任抛盘**：稳定币冻结、仲裁争议曝光、多签暂停被滥用→用户不卖币而「卖信任」→用户流失、TVL 下滑；缓解=Pause 滥用防护、透明度、冲突矩阵写死，落 Runbook §6/§11。**关键结论**：所有币都会被抛；关键不是「会不会抛」而是**抛时系统会不会死**；不发币则无币抛盘，仅有上述两类协议/信任风险，须写死缓解落点。

**控制权绝对封顶证明**：多签权限矩阵 + Immutable Core/Proxy 边界 + 无终极 override 改变资金释放规则 + 紧急多签可验证触发条件，四者共同构成「控制权绝对封顶」的写死证明；落 08-4 第 2 章、Runbook §7、01/02。

## 企业级审计分层与最终五问（必答写死）

以下**八层**对应企业级审计的链上/时间/经济/治理/公司/终局/数据/叙事；**五问**为最终审计必答；**六项证明**为须能出具的写死落点。未写死则审计结论会写 **governance-controlled / process-based / 非 immutable settlement** 等定性。

### 八层写死要求（摘要）

| 层 | 审计焦点 | 须写死落点 |
|----|----------|------------|
| **一、链上层（Protocol Hard Layer）** | Escrow 资金路径是否数学封顶 | 是否存在 admin override / upgrade hook / emergency withdraw / delegatecall 可换逻辑；**哪些逻辑不可升级、哪些升级须 timelock、哪些须 N-of-M 多签+timelock、是否允许改变历史订单逻辑**；否则=可升级金融系统非不可篡改协议。落 01/02、Runbook §7。 |
| **二、时间完整性层** | 时间一致性攻击 | 区块时间 vs 服务器时间冲突、NTP 劫持、**reorg 后 disputeWindow 是否重算**、不同时区证据冲突；**时间优先级矩阵、时间冲突 fallback 顺序、time rollback detection**；否则裁决可被时间操纵。落 08-3、Runbook §11。 |
| **三、经济层** | 仲裁员激励 + 默认裁决击穿 | 仲裁员不按件计费、无「裁得越多赚得越多」、无偏向某类结果提收益；**默认裁决击穿模拟**（如 80% 证据不足+退款优先时导游经济是否崩）、**质押覆盖率压力测试**；否则灰色证据比例高时经济性坍塌。落 08-3、Runbook §8/定稿表。 |
| **四、治理层** | 治理捕获风险 | 治理币谁初始分配、公司是否持多数、可否改仲裁规则/资金终态；**治理可改什么、永远不可改什么、是否存在 constitutional guard**；若 governance 可改资金终态=证券+可控制金融产品。落 08-4 本节、治理文档。 |
| **五、公司穿透层** | 实质控制 | 公司是否具备实际收益/仲裁/暂停/升级控制能力；**3/5 多签是否均为公司的人**（是则门限为形式）；多签是否跨法人、是否控制集中、是否公开披露结构。落 08-4 第 2 章、Runbook §10。 |
| **六、终局能力层** | 公司消失后是否自然终局 | 公司消失后**仲裁还能运行吗、Escrow 是否自动终局、存量订单是否自动清算、用户是否有自助路径**；若**必须依赖运营团队**=你不是协议、是 SaaS。落 Runbook §6、08-4 协议死亡场景。 |
| **七、数据层** | 证据链攻击 | 证据是否可批量删除、Hash 是否可被 GDPR 删除要求覆盖、司法命令是否 override 区块证据、**存储是否集中在单一云（若 S3 被封仲裁能否进行）**。落 08-4 第 3 章、Runbook §11。 |
| **八、叙事一致性层** | 叙事风险 | 对外称「去中心化旅行协议」但内部多签控制、仲裁控制、升级控制、参数可改；**若叙事 > 实际控制权下放程度**=监管最易抓的点。须写死当前=混合治理态、非完全去中心化。落 08-4 第 1/8 章。 |

### 企业级最终审计五问（必须能答）

1. **是否存在任何路径可以改变资金终态？** 答 NO 或写死唯一例外并封顶；否则=事实控制结构。
2. **是否存在任何路径可以操纵时间判定？** 答 NO 或写死时间优先级/fallback/回滚检测；否则裁决可被时间操纵。
3. **是否存在任何路径可以通过治理或多签改变规则？** 须写死治理可改/不可改、constitutional guard；否则=governance-controlled。
4. **经济模型在灰色地带比例上升时是否崩溃？** 须有默认裁决击穿模拟与质押覆盖率压力测试；否则=经济安全未闭合。
5. **公司层面是否实质控制？** 须写死多签跨法人、无控制集中、或披露；否则门限为形式。

### 六项证明（须能出具写死落点）

- ⚠ **协议数学不可变性证明**：Escrow 无 admin override/emergency withdraw/可换逻辑；哪些不可升级、哪些须 timelock、哪些须 N-of-M+timelock；是否允许改历史订单逻辑。落 01/02、Runbook §7。
- ⚠ **时间攻击完整模型**：时间优先级矩阵、区块 vs 服务器冲突、NTP 劫持、reorg 后 disputeWindow、时区证据冲突、time rollback detection。落 08-3、Runbook §11。
- ⚠ **经济击穿模拟**：默认裁决击穿模拟、质押覆盖率压力测试、灰色证据比例高时是否坍塌。落 08-3、Runbook 定稿表。
- ⚠ **治理捕获风险**：治理可改/不可改、constitutional guard、治理币分配与公司持股。落 08-4、治理文档。
- ⚠ **公司穿透控制风险**：多签跨法人、控制集中披露、实际收益/仲裁/暂停/升级能力写死。落 08-4 第 2 章、Runbook §10。
- ⚠ **终局自然消亡能力**：公司消失后仲裁/Escrow/存量订单/用户自助路径；或明确「必须依赖运营=SaaS」。落 Runbook §6、08-4。
